# 击剑赛事数据平台产品方案

## 1. 产品定位

面向击剑家庭、教练和俱乐部的免费赛事履历与对战分析平台。

第一阶段不做“官方平台替代品”，而是做公开赛事成绩的结构化检索、运动员履历整理和赛后分析工具。平台价值来自数据整理、历史追踪、对战分析、成长报告和俱乐部管理，而不是单纯搬运原始成绩。

## 2. 核心目标

- 让家长能免费、方便地查询孩子历史比赛成绩。
- 让教练能快速查看队员比赛履历和对手情况。
- 让俱乐部能沉淀本机构队员的比赛数据和成长报告。
- 在合规边界内建立可持续的数据资产。
- 后期通过广告、家庭会员和俱乐部 SaaS 商业化。

## 3. 用户角色

### 3.1 家长

核心诉求：

- 查孩子参加过哪些比赛。
- 看每场比赛名次、比分、对手和晋级情况。
- 对比同年龄组、同剑种选手表现。
- 生成可分享的成长记录。
- 收到赛事报名、成绩更新、积分变化提醒。

### 3.2 教练

核心诉求：

- 查看队员近期比赛表现。
- 快速定位输赢原因和常见对手。
- 统计队员在小组赛、淘汰赛、关键轮次的表现。
- 生成赛后复盘材料。
- 了解其他俱乐部和重点对手的表现趋势。

### 3.3 俱乐部管理者

核心诉求：

- 查看本俱乐部参赛人数、获奖情况、晋级率。
- 维护队员名单。
- 管理教练和队员关联。
- 对外展示俱乐部比赛成果。
- 生成周期性成绩报告。

### 3.4 普通访客

核心诉求：

- 搜索赛事和成绩。
- 查看基础排名。
- 了解选手公开成绩。

## 4. 合规与数据边界

### 4.1 允许采集的数据

第一阶段只采集公开赛事结果页中与比赛事实直接相关的数据：

- 赛事名称
- 举办城市
- 比赛时间
- 项目名称
- 年龄组
- 性别组
- 剑种
- 运动员姓名
- 俱乐部名称
- 名次
- 对阵轮次
- 对阵双方
- 比分
- 晋级阶段
- 积分或排名
- 数据来源 URL
- 采集时间

### 4.2 禁止采集的数据

不得采集或存储以下数据：

- 身份证件
- 手机号
- 家庭住址
- 出生日期原值
- 订单和支付信息
- 报名表单详情
- 会员中心个人资料
- 非公开接口数据
- 需要绕过权限或破解 token 才能获取的数据

### 4.3 未成年人保护策略

- 默认只展示比赛事实，不展示过度画像。
- 支持监护人认领运动员主页。
- 支持隐藏运动员主页。
- 支持删除扩展分析数据。
- 支持纠错和同名合并申诉。
- 公开页不展示敏感个人信息。
- 隐私政策中明确数据来源、用途、保留周期和删除通道。

## 5. MVP 范围

MVP 的目标是验证数据可获得性、数据质量和用户价值，不急于做完整商业系统。

### 5.1 必做功能

#### 赛事导入

- 支持输入官方赛事结果链接。
- 自动解析赛事 ID、项目 eventCode。
- 拉取项目列表、排名、对阵和比分。
- 记录数据来源和采集时间。
- 支持重复导入时自动去重。

#### 赛事列表

- 展示已收录赛事。
- 支持按时间、城市、剑种、年龄组筛选。
- 支持按赛事名称搜索。

#### 运动员搜索

- 支持按姓名搜索。
- 搜索结果展示姓名、俱乐部、剑种、年龄组、参赛次数。
- 对同名运动员给出区分信息。

#### 运动员主页

- 展示基础信息：姓名、俱乐部、主要剑种、年龄组。
- 展示历史参赛时间线。
- 展示最好名次、参赛次数、进入淘汰赛次数。
- 展示最近比赛表现。

#### 对战分析

- 展示所有对阵记录。
- 统计总胜负、胜率、总得失剑、净胜剑。
- 展示常见对手。
- 支持按赛事、轮次、时间筛选。

#### 成绩详情

- 展示某场赛事的项目排名。
- 展示小组赛和淘汰赛对阵路径。
- 支持从排名跳转到运动员主页。

#### 隐私和纠错入口

- 运动员主页提供“认领/隐藏/纠错”入口。
- 表单记录申请人、联系方式、申请类型、说明。
- 后台可审核处理。

### 5.2 暂不做功能

- 在线支付
- 复杂会员体系
- 手机 App
- 自动全网大规模爬取
- 非公开会员数据同步
- 教练训练计划
- 即时通讯

## 6. 第一版页面结构

### 6.1 首页

首页即工具入口，不做营销长页。

核心模块：

- 搜索框：搜索运动员、赛事、俱乐部。
- 最近收录赛事。
- 热门年龄组和剑种入口。
- 平台数据概览：赛事数、运动员数、对阵数。

### 6.2 赛事页

核心模块：

- 赛事基本信息。
- 项目切换。
- 排名表。
- 对阵路径。
- 俱乐部表现。
- 数据更新时间。

### 6.3 运动员页

核心模块：

- 运动员概览。
- 参赛时间线。
- 对战统计。
- 常见对手。
- 成绩趋势。
- 隐私/纠错入口。

### 6.4 对手对比页

核心模块：

- 两名运动员历史交手。
- 各自近期比赛表现。
- 胜负和比分分布。
- 共同对手。

### 6.5 俱乐部页

核心模块：

- 俱乐部队员列表。
- 近期参赛记录。
- 名次分布。
- 奖牌和前八统计。
- 年龄组和剑种分布。

### 6.6 后台管理页

核心模块：

- 数据导入任务。
- 数据质量检查。
- 隐私/纠错申请。
- 运动员合并。
- 俱乐部名称归一。

## 10. 技术路线

### 10.1 前端

建议使用：

- Vue 3 或 React
- TypeScript
- 响应式 Web，优先移动端体验
- 图表使用 ECharts

### 10.2 后端

建议使用：

- Node.js / NestJS 或 Python / FastAPI
- PostgreSQL
- Redis 队列
- 定时导入任务
- 对外提供只读 API

### 10.3 数据采集

策略：

- 第一阶段手动输入赛事链接触发导入。
- 第二阶段增加定时增量检查。
- 第三阶段建立赛事目录和自动发现机制。

原则：

- 限速请求。
- 保留来源。
- 只采集公开结果。
- 不绕过权限。
- 失败可重试。
- 接口变化有告警。

## 11. 开发阶段规划

### 阶段 0：数据验证

周期：1-2 周

目标：

- 验证 20-50 个赛事链接能否稳定解析。
- 确认排名、对阵、比分字段完整度。
- 评估同名运动员和俱乐部名称清洗难度。

交付：

- 数据导入脚本
- 数据表结构
- 样本数据报告

### 阶段 1：MVP

周期：3-5 周

目标：

- 上线可用 Web 版本。
- 支持赛事导入、运动员搜索、对战分析。
- 支持隐私/纠错申请。

交付：

- 首页
- 赛事页
- 运动员页
- 对战分析页
- 后台导入页

### 阶段 2：内测

周期：4-8 周

目标：

- 邀请 20-50 个家庭和 3-5 个教练试用。
- 收集搜索、留存、分享和反馈数据。
- 修复数据错误和体验问题。

交付：

- 反馈后台
- 运动员认领
- 隐私处理流程
- 基础数据质量报表

### 阶段 3：商业化试点

周期：8-12 周

目标：

- 测试家庭会员和俱乐部版。
- 接入广告位。
- 推出成长报告。

交付：

- 会员权益
- 报告导出
- 俱乐部看板
- 支付能力

## 12. 关键风险和应对

### 12.1 数据源变化

风险：

官方接口参数、登录方式或字段结构变化。

应对：

- 建立导入适配层。
- 保留原始响应快照。
- 数据采集任务增加异常告警。
- 不把单一接口写死在业务逻辑里。

### 12.2 合规风险

风险：

未成年人信息、公开数据再利用、商业化展示引发投诉。

应对：

- 做最小化展示。
- 建立隐藏和删除机制。
- 上线前准备隐私政策和用户协议。
- 对认领用户补充监护人确认。

### 12.3 数据质量风险

风险：

同名运动员混淆、俱乐部名称不一致、比赛阶段解析错误。

应对：

- 建立人工纠错后台。
- 使用姓名 + 俱乐部 + 剑种 + 年龄组 + 参赛轨迹做消歧。
- 所有自动合并保留可回滚记录。

### 12.4 商业风险

风险：

用户只查一次成绩，不愿长期付费。

应对：

- 提供持续价值：提醒、成长报告、对手追踪、教练复盘。
- 会员重点放在家庭长期记录和教练效率，而不是基础查询。

## 13. 第一版成功标准

数据标准：

- 能稳定导入 50 个赛事项目。
- 对阵数据完整率达到 90% 以上。
- 同名误合并率可控并可人工修正。

用户标准：

- 家长能在 30 秒内找到孩子成绩。
- 单个运动员主页能展示完整比赛时间线。
- 对战分析能回答“最近输给谁、赢过谁、差距多少”。

商业标准：

- 内测用户愿意收藏或再次访问。
- 教练愿意用它做赛后复盘。
- 家长愿意分享孩子成长页。

## 14. 下一步建议

下一步应进入数据验证阶段，先不做完整 UI。

建议优先完成：

1. 收集 20 个官方赛事结果链接。
2. 抽取接口字段，确认排名和对阵数据结构。
3. 建立 PostgreSQL 表结构。
4. 写第一版导入脚本。
5. 用样本数据生成第一个运动员主页原型。
